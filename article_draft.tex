\begin{abstract}
% TODO: write later. One-liner for now:
Lemma synthesizes per-query native executors with an LLM, then verifies them
against a deterministically transpiled Verus \texttt{MethodSpec} before compile
and reuse via a DuckDB extension.

\section{Introduction}
%% =====================================================================
% Keep short. Expand prose later.

\begin{itemize}
  \item General-purpose OLAP engines pay for generality (interpretation, operators, abstractions).
  \item LLM-synthesized per-query code can be much faster (CodexDB, GenDB, Bespoke OLAP) but is hard to trust.
  \item \textbf{Lemma:} SQL $\to$ formal \texttt{MethodSpec} $\to$ agent writes optimized \texttt{run\_query} $\to$ \textbf{Verus proves} \texttt{run\_query} $\equiv$ \texttt{MethodSpec} $\to$ native binary, cached in DuckDB.
  \item Target: recurring analytical queries where verify+compile amortizes over many runs.
\end{itemize}

\section{Pipeline}
%% =====================================================================

\begin{figure}[t]
  \centering
  % \includegraphics[width=\linewidth]{figures/pipeline}
  \fbox{\parbox{0.95\linewidth}{\centering\vspace{1.2em}
  SQL $\to$ transpile \texttt{MethodSpec}/\texttt{valid\_cols} $\to$ agent \texttt{run\_query}\\
  $\to$ assemble $\to$ Verus verify $\to$ \texttt{verus --compile} $\to$ binary\\
  $\to$ DuckDB extension cache / invoke
  \vspace{1.2em}}}
  \caption{Lemma end-to-end pipeline (Verus path).}
  \Description{Flow from SQL through MethodSpec, agent run\_query, Verus verify and compile, to DuckDB-cached binary.}
  \label{fig:pipeline}
\end{figure}

\noindent\textbf{Steps.}
\begin{enumerate}
  \item \textbf{Transpile.} Deterministic SQL$\to$Verus: columnar \texttt{Cols}, \texttt{valid\_cols} (row/cell bounds), \texttt{method\_spec} / \texttt{method\_spec\_helper}. Schema-driven (no hard-coded benchmark columns).
  \item \textbf{Agent.} Writes only the \texttt{run\_query} body (optimized loop / agg). Host injects signature + \texttt{requires valid\_cols} + \texttt{ensures} $\equiv$ \texttt{method\_spec}. Agent must not add contracts.
  \item \textbf{Assemble.} Single \texttt{.rs}: spec + proved body + trusted arith/agg helpers + \texttt{load\_cols} + timing \texttt{main}.
  \item \textbf{Verify.} Verus/Z3: \texttt{run\_query} exec $\equiv$ \texttt{method\_spec}. Fail $\Rightarrow$ reject (no ship).
  \item \textbf{Compile.} \texttt{verus --compile} $\to$ native binary (no Dafny postprocessor path).
  \item \textbf{Serve.} DuckDB extension caches binary; repeat queries invoke native path.
\end{enumerate}

\noindent\textbf{Trust boundary (concise).}
\begin{itemize}
  \item \textbf{Proved:} agent \texttt{run\_query} $\equiv$ \texttt{method\_spec} (scalar + group-by fixtures; ghost map tied to helper in same loop as exec agg).
  \item \textbf{Trusted:} wrapping arith under \texttt{valid\_cols}; NativeAgg-style \texttt{hashmap\_*}/\texttt{agg\_*}; \texttt{load\_cols} I/O; some SQL-feature helpers (LIKE, certain joins/set ops) when used.
  \item \textbf{Trusted transpiler:} SQL $\equiv$ \texttt{method\_spec} checked empirically / tests, not proved in Verus.
  \item Residual gap same class as Dafny externs: malicious \texttt{external\_body} can lie; agent-written proved body cannot.
\end{itemize}

\noindent\textbf{Why Verus (vs prior Dafny path).}
\begin{itemize}
  \item One artifact: annotated Rust is both proof subject and executable (no translate + unsafe peephole pass).
  \item Avoids Dafny\texttt{Int}/BigInt vs bitvector timeout tradeoff and postprocessor soundness hole.
  \item Hot path is ordinary Rust (\texttt{HashMap}, slices) with ghost state for the proof.
\end{itemize}

\subsection{DuckDB extension}
\begin{itemize}
  \item \texttt{SELECT lemma('SQL...')} triggers optimize / verify / compile on miss; cache hit runs native binary.
  \item WIP polish; research harness also runs standalone (\texttt{benchmark\_verified.py}).
\end{itemize}

\subsection{Agent sandbox (extension)}
\begin{itemize}
  \item Production need: sandbox the optimizer agent (or at least its tool calls). Status as of 2026-07-21: host OpenRouter ReAct loop; tools run in Docker (\texttt{--network none} by default). Not fully ``production-hardened'' yet.
  \item Inspiration: Inspect AI tool-call sandboxing; Inspect agent bridge as a way to let users plug in their own agent CLI.
  \item Data policy modes for DuckDB tool: none / stats / full (read-only).
\end{itemize}

\subsection{SQL surface (sketch)}
\begin{itemize}
  \item Analytical subset: aggs, filters, GROUP BY/HAVING, joins, subqueries/CTEs, set ops, ORDER BY/LIMIT --- many advanced shapes still use TRUSTED helpers.
  \item Fixtures: SSB flat, TPC-H (e.g.\ Q1/Q6), plus basic-SQL holdouts.
\end{itemize}

\section{Related Work}
%% =====================================================================
% Bullets only --- expand to prose later.

\paragraph{LLM-synthesized query engines.}
\begin{itemize}
  \item \textbf{CodexDB}~\cite{Trummer2022CodexDB}: LLM synthesizes per-query processing code from SQL (+ NL instructions); correctness via tests / execution, not formal proof vs a MethodSpec.
  \item \textbf{GPT-DB}~\cite{Trummer2023GPTDB}: follow-on demo; customizable GPT-generated SQL processing code.
  \item \textbf{GenDB}~\cite{Lao2026GenDB}: multi-agent instance-optimized C++ per query (workload / storage / plan / codegen / optimize); strong speedups vs DuckDB/Umbra; correctness mainly empirical / inspection.
  \item \textbf{Bespoke OLAP}~\cite{Wehrstein2026Bespoke}: full workload-specific C++ engine (storage + executors) with iterative perf feedback; validates vs DuckDB, not formal refinement.
\end{itemize}

\paragraph{Classical query compilation (not LLM).}
\begin{itemize}
  \item \textbf{HIQUE}~\cite{Krikellas2010HIQUE}: template-based holistic per-query C codegen.
  \item \textbf{HyPer}~\cite{Neumann2011Hyper} / adaptive execution~\cite{Kohn2018Adaptive}: data-centric LLVM compilation; interpret$\leftrightarrow$compile switching.
  \item \textbf{LegoBase}~\cite{Klonatos2014LegoBase}: generative programming / staging; Scala$\to$specialized C.
  \item \textbf{Umbra}~\cite{Kersten2021UmbraFlyingStart}: fast compile (Flying Start) + optimizing LLVM path.
  \item \textbf{NoisePage / Peloton line}~\cite{Menon2017Relaxed,Menon2020Permutable}: relaxed operator fusion; permutable compiled queries without full recompile.
  \item \textbf{DBToaster}~\cite{Ahmad2012DBToaster}, \textbf{Tupleware}~\cite{Crotty2015Tupleware}, \textbf{Weld}~\cite{Palkar2018Weld}: specialized / IR-based codegen lineages (cite selectively).
\end{itemize}

\paragraph{Formal SQL semantics \& verified query compilers.}
\begin{itemize}
  \item \textbf{HoTTSQL}~\cite{Chu2017HoTTSQL}: mechanized denotational SQL semantics (Coq); prove rewrite rules.
  \item \textbf{Cosette}~\cite{Chu2017Cosette}: automated SQL equivalence prover (proof or counterexample).
  \item \textbf{SQL$_{\mathrm{Coq}}$ / SQL algebra}~\cite{Benzaken2019SQLCoq}: Coq semantics for realistic SQL + algebra equivalence.
  \item \textbf{Physical algebra / QEP layer}~\cite{Benzaken2018Physical}: Coq formalization of low-level RDBMS operators / physical plans.
  \item \textbf{Q$^\ast$cert}~\cite{Auerbach2017Qcert}: Coq platform for verified query compilation (NRA$^e$, multi-target).
  \item \textbf{DBCert}~\cite{Benzaken2022DBCert}: verified SQL$_{\mathrm{Coq}}$ $\to$ Imp (then JS); correct-by-construction compiler, \emph{not} LLM-written hot paths.
  \item Contrast w/ Lemma: those systems verify a \emph{fixed} compiler or rewrite; Lemma verifies an \emph{LLM-written} executor against a transpiled MethodSpec.
\end{itemize}

\paragraph{LLM + machine-checked code (general, not DB-specific).}
\begin{itemize}
  \item \textbf{AlphaVerus}~\cite{Aggarwal2025AlphaVerus}: LLM$\to$verified Verus via translation + verifier-guided tree search (``Treefinement'').
  \item \textbf{Vericoding benchmarks}~\cite{Dougherty2025Vericoding}: large Dafny/Verus/Lean specs for LLM verified synthesis eval.
  \item Related: Formal Disco / verified-program datasets; DafnyBench-style eval --- cite if space.
\end{itemize}

\paragraph{Adjacent LLM$\times$DB (not executor synthesis).}
\begin{itemize}
  \item \textbf{SQLBarber}~\cite{Lao2025SQLBarber}: LLM generates customized SQL \emph{workloads} / templates (benchmarking), not query executors.
  \item Text-to-SQL / NL$\to$SQL systems: large literature; orthogonal (query \emph{text}, not verified runtime).
\end{itemize}

\paragraph{Verification substrate.}
\begin{itemize}
  \item \textbf{Verus}~\cite{Lattuada2023Verus}: SMT verification for Rust; Lemma's proof target.
  \item \textbf{Dafny}~\cite{Leino2010Dafny}: prior Lemma path; same agent-vs-spec idea, heavier codegen/postprocess story.
\end{itemize}

\paragraph{Positioning (one line).}
\begin{itemize}
  \item Lemma $\approx$ GenDB/CodexDB-style synthesis \emph{plus} DBCert/Cosette-style machine-checked guarantees, scoped to \texttt{run\_query} $\equiv$ MethodSpec before ship.
\end{itemize}

\subsection{Holdout microbenchmark (agent-style kernels)}
\label{sec:holdout-eval}

\noindent\textbf{Snapshot.}
Measured \texttt{2026-07-21T22:38:52Z} (UTC); holdout artifact commit \texttt{dcc9d59}.
Fair protocol: both sides
start from \emph{already loaded} in-memory columns / DuckDB tables; 3 warmups then
median of 5 timed query-only runs. Lemma kernels written as a perfect agent would
(zone maps at ingest, dict codes at ingest, in-place hash probe, dense small-card
buckets); query predicates / join build+probe / aggs stay on the clock.

\noindent\textbf{Hardware (this machine).}
AMD Ryzen AI 5 330 (8 logical CPUs, SMT), $\sim$7.8\,GiB RAM, Linux WSL2.
MT modes: \texttt{RAYON\_NUM\_THREADS=8}, DuckDB \texttt{PRAGMA threads=8}.

\noindent\textbf{Takeaways (this snapshot).}
Lemma ST $\le$ DuckDB 1T on 24/25 queries (H15 $\approx$ tie). Wins concentrate on
zone-pruned scans, dict-coded filters, small-card group-by, and Zipf joins; MT
helps when work is large enough (e.g.\ H15/H17/H25) and hurts tiny kernels.
Scale-up gaps (partitioned join build, vectorized scans, spill hash, DuckDB-backed
columnar load) are tracked as flagged primitives / \texttt{verus/db\_extension}
experiments---not claimed solved by this table.

\begin{itemize}
  \item Additional baselines still planned: full Verus-proved binaries (not only
        agent-primitive holdout kernels), verify+compile amortization, cache hits.
  \item Larger GCP / SF-scale runs deferred; protocol above is the intended fair compare.
\end{itemize}

\section{Future Work}
%% =====================================================================

\begin{enumerate}
  \item Shrink TRUSTED surface: extend Verus coverage so low-level perf ops need fewer \texttt{external\_body} externs.
  \item Hardware-aware optimization: give agent HW info; needs verification support for HW-specific objects/params.
  \item Stronger optimizing harness / RSI-style self-improvement loop (cite recent RSI harness papers --- TODO).
  \item Distributed executors: interesting, but verification challenges multiply (networks, faults, consistency).
  \item Harden extension sandbox; optional Inspect-style / pluggable agent CLI bridge.
\end{enumerate}
